# Ultra Brainstorm — 統合ブレインストーミング

アイデアを完全な設計文書へ発展させるスキル。

> **⛔ 設計承認ゲート（例外なし）**
>
> 設計が承認されるまで、コードの実装・スキャフォールド・ファイル作成を一切行ってはならない。
> 「シンプルだから設計は不要」は許されない。全プロジェクトに適用。
>
> **例外**: typo修正、1行のバグ修正、依存関係更新など明らかに設計不要な変更は、brainstormをスキップして直接 ultra-debug / ultra-implement に進んでよい。

## 鉄則

- **1回に1つの質問** — 複数質問を同時に投げない
- **選択式を優先** — オープンエンドより答えやすい
- **YAGNI厳守** — 不要な機能は設計から排除

## プロセス

### Step 1: コンテキスト探索

プロジェクトの現状を完全把握する：

```text
# ファイル構造確認
list_dir / mcp_filesystem_list_directory → プロジェクトルート
view_file → package.json / Cargo.toml / go.mod

# 最近のコミット
run_command: git log --oneline -10

# 既存ドキュメント
view_file → README.md
```

### Step 2: Search Before Building（三層知識探索）

何かを作る前に、既存の解決策を検索する：

**Layer 1 — 定番（検証済み）:**
- 同じことを実現するライブラリ/フレームワークは存在するか？
- 車輪の再発明を避ける。ただし常識を疑う余地は残す。
- Context7 で関連ライブラリのベストプラクティスを確認:
  ```yaml
  mcp_context7_resolve-library-id: [関連ライブラリ]
  mcp_context7_query-docs: [実現したい機能のベストプラクティス]
```

**Layer 2 — 新興（人気上昇中）:**
- 新しいトレンドやアプローチの検索
- ただし流行に飛びつかず批判的に評価する

**Layer 3 — 第一原理:**
- この具体的な問題に対する最適解を原理から導く
- 既存の常識が間違っている可能性を検討する
- **EUREKA moment:** 常識に反する洞察を発見した場合は明示する:

```yaml
🔬 EUREKA: [従来の常識] は [この文脈では] 誤りの可能性がある。
   根拠: [第一原理からの説明]
   推奨: [新しいアプローチ]
```

### Step 3: 明確化質問（4段構造）

ユーザーに質問する際は以下の構造で:

```yaml
1. 再接地: 「現在の理解では [X] ですが、合っていますか？」
2. 平易化: 技術用語を避け、要点を1文で
3. 推奨:  「[Y] を推奨します。理由は [Z] です」
4. 選択肢: 「A か B、どちらが良いですか？」（具体的な2択以上）
```

- 1つの質問 → 回答 → 次の質問（1つずつ順番に）
- スコープが大きすぎる場合は分解を提案

### Step 4: アプローチ提案

2〜3つのアプローチを提案：

```text
## アプローチ A（推奨）
- 概要: ...
- メリット: ...
- デメリット: ...
- 完全性スコア: 9/10

## アプローチ B
- 概要: ...
- メリット: ...
- デメリット: ...
- 完全性スコア: 6/10
```

**完全性原則（Boil the Lake）:**
- AIで実装コストが低い場合、常に完全版（スコア高い方）を推奨
- 「80%で十分」は旧来思考。AIでは100%が数分多いだけ

### Step 4.5: プロジェクト初期ファイル生成（新規PJのみ）

新規プロジェクトの場合、設計文書作成前にプロジェクト基盤ファイルを生成：

```bash
# .gitignore（node_modules等の誤追跡を防止）
run_command: |
  cat > .gitignore << 'EOF'
  node_modules/
  dist/
  .env
  .env.local
  .DS_Store
  *.log
  coverage/
  EOF
```

**重要**: `.gitignore` は初回コミット前に必ず作成すること。後から追加しても
既にステージされたファイルは追跡から除外されない（`git rm --cached` が必要）。

検出ルール:
- `package.json` が存在しない → 新規PJと判断
- `.gitignore` が存在しない → 自動生成を提案
- `.git/` が存在しない → `git init` も提案

### Step 5: 設計文書作成

承認されたアプローチを文書化：

```bash
run_command: mkdir -p docs/specs
```

`docs/specs/YYYY-MM-DD-<topic>-design.md` に保存：

- アーキテクチャ概要
- コンポーネント設計
- データフロー
- エラー処理
- テスト方針（unit / integration / e2e の比率方針。具体的テストケースは ultra-plan で定義）

### Step 6: 設計承認 → ultra-plan へ

> 「設計文書を `<path>` に保存しました。確認後、実装計画の作成に進みます。」

承認されたら **ultra-plan** スキルに遷移。
