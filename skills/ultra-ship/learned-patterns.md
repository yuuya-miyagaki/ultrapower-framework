### 本番デプロイ前のセキュリティルールテスト必須化（自動検出: 2026-03-25）

**状況**: Firestore セキュリティルールを本番環境で直接テストした結果、サブコレクションの権限漏れを3回のデプロイサイクルを費やして修正した。
**解決策**: デプロイ前に必ずエミュレータでルール検証を行う。

```bash
# デプロイ前チェックリスト（Firebase プロジェクト）
# 1. ルールのコンパイルチェック（MCP ツールで即時）
mcp_firebase-mcp-server_firebase_validate_security_rules

# 2. エミュレータでのルールテスト
firebase emulators:start --only firestore
# テストスイートを実行

# 3. テスト通過後にルールのみデプロイ
firebase deploy --only firestore:rules

# 4. 全体デプロイ
firebase deploy
```

**教訓**: 「セキュリティルールはコンパイルチェックだけでは不十分」— ルールがコンパイルに通っても、実際のデータパスで期待通りに動作するかは別問題。

### プロトタイプ/MVP でもテストは必須（自動検出: 2026-03-25）

**状況**: 「素早いプロトタイプ」としてTDDをスキップし、テストカバレッジ3.7%の状態で本番デプロイ。結果として権限エラーが本番でのみ発覚。
**解決策**: 本番デプロイするコードには必ずテストを書く。TDD適用除外はCSS/設定ファイル/ドキュメントのみ。

```text
判断フロー:
  このコードは本番にデプロイする？
  ├── YES → テスト必須（例外なし）
  │   ├── セキュリティルール → エミュレータテスト
  │   ├── API統合 → 統合テスト
  │   └── UI → E2Eテスト or スクリーンショット比較
  └── NO → ローカル実験のみ → テスト任意
```

**教訓**: テストをスキップして節約した時間（推定30分）よりも、本番デバッグに費やした時間（推定2時間）の方が長かった。
