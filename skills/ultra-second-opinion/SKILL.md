# Ultra Second Opinion — 二次意見ブリッジ

> gstack `/codex` の代替アプローチ。Antigravity（Gemini）から別のAIに直接依頼する仕組み。

## コンセプト

**なぜ二次意見が重要か：**

1つのAIだけに頼ると、そのAI特有のバイアスや盲点がそのまま製品に入る。
AntigravityはGeminiベース。異なるAI（Claude Code, ChatGPT, Copilot等）に聞くことで**異なる視点**を得られる。

**この仕組み：**

Antigravityが現在のコンテキスト（コード変更、計画、問題）を分析して、対象AIに渡す最適なプロンプトを生成。ユーザーはコピペするだけ。

## 起動条件

> **核心ルール: 「やり直しコストが高い決定」の直前で発動する**
>
> 小さな修正に毎回セカンドオピニオンを取ると開発速度が落ちる。
> 逆に、DB選定やセキュリティ設計でスキップすると後で数日失う。
> **不可逆性の高さ** で判断する。

### 🔴 必ず発動（高コスト・不可逆な決定）

| 状況 | 理由 | 連携スキル |
|------|------|-----------|
| 不可逆な技術決定の直前 | DB選定・認証方式・デプロイ先など。後から変更するコストが巨大 | ultra-plan |
| セキュリティルールの変更時 | 1つのAIの見落とし＝脆弱性。攻撃面は複数の目で見るべき | ultra-ship |
| 3回失敗でエスカレーション時 | 私が解決できない問題を同じ私に聞いても無意味。別AI必須 | ultra-debug |

→ これらの状況では**ユーザーに確認せず自動的に提案**する。

### 🟡 提案すべき（判断はユーザーに委ねる）

| 状況 | 理由 | 連携スキル |
|------|------|-----------|
| Phase／設計の開始前 | 設計ミスは後工程で10倍のコスト。最安のチェックポイント | ultra-brainstorm |
| ultra-review 完了後 | レビューのダブルチェック。ただし毎回だと重いのでユーザー判断 | ultra-review |
| 本番デプロイ前（初回） | 最後の砦。特に初回デプロイ時 | ultra-ship |

→ 「セカンドオピニオン取りますか？」と提案。ユーザーが判断。

### ⚪ 不要（過剰・コスト対効果が悪い）

| 状況 | 理由 |
|------|------|
| 日常的なバグ修正 | コスト対効果が悪い |
| CSS／スタイル変更 | 別AIに聞く価値がない |
| テスト追加 | テスト自体が検証手段 |
| ドキュメント更新 | ultra-docs の一貫性チェックで十分 |

→ これらの状況では発動しない。

### 明示的トリガー（ユーザーからの直接指示）

以下のキーワードで常に即座に起動:

- 「二次意見」「セカンドオピニオン」「別の意見が欲しい」
- 「Claude に聞いて」「ChatGPT で確認」「別のAIで確認」

## Step 1: 対象AI選択

ユーザーに確認:

```text
どのAIに二次意見を求めますか？
  A) Claude Code（ターミナル）
  B) ChatGPT（ブラウザ）
  C) GitHub Copilot（エディタ）
  D) その他（指定してください）
```

### AI別の前提確認

| AI | 確認事項 | インストール |
|----|----------|-------------|
| Claude Code | `claude` コマンドが使えるか | `npm install -g @anthropic-ai/claude-code` |
| ChatGPT | ブラウザでアクセス可能か | https://chatgpt.com |
| Copilot | エディタ拡張が有効か | VS Code / JetBrains 拡張 |

---

## Step 2: モード選択

ユーザーの意図に応じてモードを自動判定:

| 意図 | モード | 説明 |
|------|--------|------|
| 「コードレビューして」 | Review | diffの問題点を検出 |
| 「壊してみて」「攻撃して」 | Challenge | 攻撃者視点でコードを分析 |
| 「設計をレビュー」 | Plan Review | 実装計画の穴を探す |
| 「アーキテクチャを見て」 | Architecture | スケーラビリティ・設計判断を評価 |
| 「自由に質問」 | Consult | なんでも質問 |

---

## Step 3: コンテキスト収集

モードに応じて必要な情報を自動収集:

### Review / Challenge モード

```bash
# デフォルトブランチ名を動的検出
run_command: git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's@^refs/remotes/origin/@@' || echo "main"

# 変更されたファイルを確認（<default> は上記で検出したブランチ名）
run_command: git diff <default> --stat
run_command: git diff <default> --name-only
run_command: git log <default>..HEAD --oneline
```

変更内容を分析して、レビューで注目すべきポイントを特定。

### Plan Review モード

実装計画ファイルを読み取り、レビューポイントを整理。

### Architecture モード

```text
# プロジェクト構成を把握
find_by_name: Extensions=["ts", "tsx", "js", "jsx"], SearchDirectory="src/"
view_file: package.json（先頭20行）
```

## Step 4: プロンプト生成

**コンテキストを織り込んだ最適なプロンプト**を生成し、ユーザーにコピペ用として提示。

### Review モード

```text
╔══════════════════════════════════════════════════════════╗
║  📋 [対象AI名] にコピペしてください                        ║
╠══════════════════════════════════════════════════════════╣

  ターミナルで以下を実行:

  cd [プロジェクトパス]

  claude "[生成されたプロンプト]"

╚══════════════════════════════════════════════════════════╝
```

プロンプトの生成ルール:

1. **日本語で生成**（ユーザーが日本語話者のため）
2. **具体的に指示**（「コードを見て」ではなく「git diff mainの変更を見て、セキュリティ問題・バグ・パフォーマンス問題を深刻度P1-P3で指摘して」）
3. **厳しさを指定**（「褒めなくていいので問題だけ教えて」）
4. **変更内容を要約**（対象AIが何を見ればいいかわかるように）

### 各モードのプロンプトテンプレート

#### Review

```bash
git diff <default> のコード変更をレビューしてください。

【主な変更内容】
{Antigravityが分析した変更の要約}

【重点チェック項目】
- セキュリティ問題（インジェクション、認証バイパス、データ漏洩）
- バグ（エッジケース、エラーハンドリング漏れ、レースコンディション）
- パフォーマンス問題（N+1クエリ、メモリリーク、不要な再レンダリング）

見つかった問題は深刻度(P1: 致命的 / P2: 重要 / P3: 改善推奨)をつけてください。
褒めなくていいので、問題だけ教えてください。容赦なくお願いします。
```

#### Challenge

```text
このプロジェクトの {対象ディレクトリ/ファイル} を攻撃者の視点でレビューしてください。

【攻撃対象の概要】
{Antigravityが分析した対象の説明}

以下の観点で脆弱性を探してください:
- インジェクションベクター（SQL, XSS, コマンドインジェクション）
- 認証バイパス
- 権限昇格
- データ漏洩
- タイミング攻撃
- リソース枯渇（DoS）

褒めなくていいです。壊せる場所を全部教えてください。
```

#### Plan Review

```yaml
以下のファイルの実装計画をレビューしてください: {計画ファイルパス}

【計画の概要】
{Antigravityが分析した計画の要約}

以下の観点で厳しくチェックしてください:
- 論理的な抜け・矛盾
- エッジケースの見落とし
- もっとシンプルなやり方がないか
- 実現可能性のリスク（何がうまくいかない可能性があるか）
- 依存関係の順序問題

褒めなくていいので、問題と改善点だけ教えてください。
```

#### Architecture

```text
このプロジェクトのアーキテクチャを評価してください。

【技術スタック】
{package.jsonやプロジェクト構成から自動検出}

【現在の構成】
{ディレクトリ構造の要約}

以下の観点で意見をください:
- 3ヶ月後・1年後のスケーラビリティ
- 現在の技術的負債
- アーキテクチャ上の懸念点
- より良い構成の提案（あれば）

率直な意見をお願いします。
```

#### Consult（自由質問）

ユーザーの質問をそのまま整形して渡す。必要に応じてプロジェクトコンテキストを追加。

## Step 5: フィードバック活用

対象AIの出力をユーザーが共有してくれた場合：

1. **対象AIの指摘を読み取る**
2. **Antigravity（私）の見解と照合**
3. **合意点と相違点を整理**

```text
╔══════════════════════════════════════════════════════════╗
║  CROSS-AI 分析                                          ║
╠══════════════════════════════════════════════════════════╣
║  両方が指摘:     [N] 件（信頼度: 高）                    ║
║  対象AIのみ:   [N] 件（検討推奨）                        ║
║  Antigravityのみ: [N] 件（検討推奨）                     ║
║  意見相違:       [N] 件（詳細下記）                      ║
╚══════════════════════════════════════════════════════════╝
```

意見が違う場合は**両方の根拠を提示**して、ユーザーが判断できるようにする。

---

## レッドフラグ

- ❌ 対象AIの出力を要約して省略する（全文を尊重）
- ❌ 「私の方が正しい」と一方的に判断する（根拠を提示して判断はユーザーに）
- ❌ コンテキストなしのプロンプトを生成する（対象AIが何を見るべきかわからない）
- ❌ ユーザーの技術レベルに合わないプロンプトを生成する（コピペで使えることが大前提）
