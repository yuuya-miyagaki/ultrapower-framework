# Ultra Docs — ドキュメント同期

> gstack /document-release を Antigravity + GitHub MCP に最適化

## 起動条件

- ultra-ship 完了後（推奨）
- 「ドキュメント更新して」「docs同期」
- PRマージ後にプロアクティブに提案

**自動化方針**: 明らかな事実更新は自動実行。リスクのある変更のみユーザーに確認。

## Step 1: 事前分析

### 1.1 ブランチ確認

```bash
run_command: git branch --show-current
```

デフォルトブランチ上の場合 → 「⚠️ デフォルトブランチ上です。ドキュメント更新用ブランチ `docs/update-[日付]` を作成しますか？」と提案。ユーザーが拒否した場合はそのまま続行を許可。

### 1.2 diff分析

以下を実行（`<default>` は動的に検出したデフォルトブランチ名）:

```bash
# デフォルトブランチ名を動的検出
run_command: git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's@^refs/remotes/origin/@@' || echo "main"

run_command: git diff <default>...HEAD --stat
run_command: git log <default>..HEAD --oneline
run_command: git diff <default>...HEAD --name-only
```

### 1.3 ドキュメントファイル発見

```yaml
mcp_filesystem_search_files: path=".", pattern="**/*.md", excludePatterns=[".git", "node_modules", "docs/generated-reports"]
```

### 1.4 変更分類

- **新機能** — 新ファイル、新コマンド、新スキル
- **変更** — 修正されたサービス、API変更
- **削除** — 削除されたファイル、機能
- **インフラ** — ビルド、テスト、CI

出力: 「N ファイル変更（M コミット）。K ドキュメントファイルをレビュー。」

## Step 2: ファイルごとのドキュメント監査

各ドキュメントファイルを読み、diffと照合:

### README.md

**README.md が存在しない場合 → テンプレート自動生成を提案:**

```markdown
# [プロジェクト名]（package.json の name / ディレクトリ名から自動取得）

[package.json の description / 1行説明を挿入]

## セットアップ

（package.json の scripts / 技術スタック情報から自動生成）

## 使い方

（主要機能の簡潔な説明）

## 開発

（テスト実行方法、ビルド手順）
```

情報ソース: `package.json`, `Cargo.toml`, `go.mod`, ディレクトリ構造から自動収集。

**README.md が存在する場合 → 通常の監査:**
- diffで追加された機能・ケイパビリティが記載されているか？
- インストール・セットアップ手順が変更と一致しているか？
- 例、デモ、使用方法の説明は有効か？
- トラブルシューティング手順は正確か？

### ARCHITECTURE.md
- ダイアグラムとコンポーネント説明がコードと一致するか？
- 設計決定の「なぜ」説明は正確か？
- **保守的に** — diffで明確に矛盾するものだけ更新

### AGENTS.md / CLAUDE.md / プロジェクト設定
- プロジェクト構造が実際のファイルツリーと一致するか？
- コマンド、スクリプトが正確か？
- ビルド/テスト手順が package.json 等と一致するか？

### その他の .md ファイル
- ファイルを読み、目的と対象者を判断
- diffと照合して矛盾を確認

### 分類

- **自動更新** — diffから明白な事実修正
- **ユーザー確認** — ナラティブ変更、セクション削除、大規模書き換え

## Step 3: 自動更新の適用

明確な事実更新を即座に適用。

各変更に1行サマリーを出力:
- ❌ 「README.md更新」
- ✅ 「README.md: スキル一覧に /ultra-benchmark を追加、スキル数を10→13に更新」

**自動更新しない対象:**
- READMEのイントロや位置づけ
- ARCHITECTURE の哲学・設計根拠
- セキュリティモデルの説明
- セクションの丸ごと削除

## Step 4: リスクのある変更のユーザー確認

各リスキーな更新に対して:
- コンテキスト説明
- 推奨: 「[X] を選択。理由: [1行]」
- オプションにはスキップを含む

## Step 5: CHANGELOG 音声ポリッシュ

> **CRITICAL: CHANGELOG エントリを上書き・置換・再生成しない**

このステップは音声のポリッシュのみ:
1. 全 CHANGELOG.md を最初に読む
2. ワーディングのみ修正。エントリの削除・並べ替え・置換禁止
3. 「ユーザーが何をできるようになったか」を先頭に
4. コミットメッセージのような文は書き換え
5. 内部向け変更は「### 開発者向け」サブセクションに分離

**このブランチで変更されていなければスキップ。**

## Step 6: クロスドキュメント一貫性チェック

個別監査後にクロスチェック:

1. READMEの機能リスト ↔ AGENTS.md の記述
2. ARCHITECTURE のコンポーネント ↔ プロジェクト構造
3. CHANGELOG の最新バージョン ↔ VERSION ファイル
4. **ディスカバラビリティ**: 全ドキュメントがREADMEからリンクで到達可能か？到達不可なら指摘
5. ドキュメント間の矛盾を検出 → 事実的矛盾は自動修正、ナラティブ矛盾はユーザー確認

## Step 7: コミットとプッシュ

### 変更なしチェック

```bash
run_command: git status --short
```

ドキュメントファイルの変更がなければ「全ドキュメントは最新です。」で終了。

### コミット

```bash
run_command: git add [変更されたドキュメントファイルを個別指定]
run_command: git commit -m "docs: プロジェクトドキュメントを更新"
run_command: git push
```

### PR本文更新（GitHub MCP使用）

```text
mcp_github_get_pull_request → PRが存在するか確認
```

PRが存在する場合、本文に `## ドキュメント` セクションを追加/更新:
- 各変更ファイルの具体的な変更内容を記載

## Step 8: 完了レポート

```yaml
ドキュメント健全性:
  README.md       [ステータス] ([詳細])
  ARCHITECTURE.md [ステータス] ([詳細])
  AGENTS.md       [ステータス] ([詳細])
  CHANGELOG.md    [ステータス] ([詳細])
  VERSION         [ステータス] ([詳細])
```

ステータス:
- 更新済 — 変更内容の説明付き
- 最新 — 変更不要
- 音声修正 — ワーディング調整のみ
- スキップ — ファイル不存在

```text
╔══════════════════════════════════════════╗
║  ULTRA-DOCS 完了                         ║
║  STATUS: [DONE / DONE_WITH_CONCERNS]     ║
╠══════════════════════════════════════════╣
║  監査ファイル:    [N] 件                 ║
║  自動更新:        [N] 件                 ║
║  ユーザー確認:    [N] 件                 ║
║  CHANGELOG修正:   [Y/N]                  ║
║  一貫性問題:      [N] 件                 ║
╚══════════════════════════════════════════╝
```

## 重要ルール

1. **読んでから編集** — 常にファイル全体を読んでから修正
2. **CHANGELOG を壊さない** — 音声ポリッシュのみ。エントリの削除・置換・再生成禁止
3. **何が変わったか明示** — 各編集に1行サマリー
4. **汎用ヒューリスティクス** — プロジェクト固有ではなく、どのリポジトリでも動く
5. **ディスカバラビリティ重要** — 全ドキュメントがREADMEから到達可能
6. **フレンドリーでユーザー目線の音声** — コードを見ていない賢い人に説明するように
