# Ultra Plan — TDD実装計画 + 3段階レビュー

> gstack plan-ceo-review + plan-eng-review + plan-design-review + Superpowers writing-plans の統合

## 起動条件

ultra-brainstorm で設計が承認された時に起動。

### レビューモード選択

| モード | 実行レビュー | 推奨場面 |
|--------|-------------|----------|
| **標準**（デフォルト） | CEO + Eng + Design の3段階 | 新機能、大規模な変更 |
| **簡易**（`--quick`） | Eng レビューのみ | 小規模変更、バグ修正後の再計画、時間制約がある場合 |
| **軽量**（`--light`） | スコープ確認 + テスト計画のみ | ミニアプリ、プロトタイプ、学習用PJ、テストプロジェクト |

簡易モードが適切な場合:
- 変更ファイル数が 5 以下
- 既存アーキテクチャへの追加のみ（新コンポーネント不要）
- UI変更なし（Design レビュー不要）

軽量モードが適切な場合:
- プロダクション用ではないプロジェクト
- 総タスク数が 10 以下
- 外部依存が少ない（セキュリティリスク低）

## Step 1: 計画生成

承認済み設計から実装計画を生成:

1. `docs/specs/` から最新の設計文書を読み込む（ultra-brainstorm が `YYYY-MM-DD-<topic>-design.md` として保存）
2. 各機能をタスクに分解（1タスク = 1テスト + 1実装）
3. タスク間の依存関係を明示
4. 各タスクに TDD サイクル指示を付与

### タスク粒度ガイド

| 粒度 | 基準 | 行動 |
|------|------|------|
| **適正** | 1テスト + 1実装ファイルで完結 | そのまま採用 |
| **大きすぎ** | 複数テストファイルにまたがる | サブタスクに分解 |
| **小さすぎ** | ワンライナーの変更 | 隣接タスクと統合 |

> 粒度基準は「テスト+実装の完結単位」。時間見積もりはプロジェクト規模に依存するため基準に含めない。

### タスクフォーマット

```markdown
## Task N: [タスク名]

### テスト（RED）
- テストファイル: [パス]
- テスト内容: [何を検証するか]

### 実装（GREEN）
- 実装ファイル: [パス]
- 実装内容: [最小限の実装]

### リファクタ
- [必要なリファクタリング]

### 検証コマンド
- `[テスト実行コマンド]`
```

### 設計文書 内部整合性チェック（計画生成後に実行）

設計文書がセクション間で矛盾していないか検証する。

> **起源**: CROSS-AI 分析（2026-03-27）で、設計文書§8「リスク軽減策」に「ネイティブ fallback」と記載しながら§5.1「UI設計」にその操作経路が存在しない矛盾が発覚。生成時にもレビュー時にも発見できなかった。

| # | チェック項目 | 確認内容 |
|---|------------|---------|
| 1 | **UI設計 ↔ リスク軽減策** | リスク軽減策で言及した UI フォールバックが、UI 設計に操作経路として存在するか |
| 2 | **データモデル ↔ スコープ** | スコープで定義した機能が必要とするフィールドが、データモデルに全て含まれるか |
| 3 | **テスト方針 ↔ UI設計** | テスト検証対象の UI 要素が、UI 設計に存在するか |
| 4 | **スコープ ↔ スコープ外** | 同じ機能がスコープ内とスコープ外の両方に記載されていないか |

矛盾を発見した場合は、計画生成前に設計文書を修正する。

## Step 2: CEO戦略レビュー（4モードスコープ管理）

### 2A. 前提チャレンジ

1. **これは正しい問題か？** 別のフレーミングでもっと簡潔な解決策はないか？
2. **何もしなかったら？** 本当の痛みか仮説的な痛みか？
3. **既存コードの活用**: 各サブ問題の何%が既存コードで解決済みか？

### 2B. Dream State Mapping

```text
  現在の状態           →  この計画     →  12ヶ月後の理想
  [describe]                [delta]          [target]
```

### 2C. Implementation Alternatives（必須）

最低2つの代替案を提示:

```yaml
APPROACH A: [名前]
  概要:   [1-2文]
  工数:   [S/M/L/XL]
  リスク: [Low/Med/High]
  利点:   [2-3項目]
  欠点:   [2-3項目]
  再利用: [既存コード/パターン]

APPROACH B: [名前]
  ...
```

- 1つは「最小実行可能版」（最少ファイル、最小差分）
- 1つは「理想アーキテクチャ版」（長期的最適）

### 2D. スコープモード選択

ユーザーに4つのモードを提示:

| モード | 説明 |
| ------ | ---- |
| **EXPANSION** | 夢大きく。10xビジョン + Platonic Ideal + Delight機会 |
| **SELECTIVE** | ベースライン維持 + チェリーピック。中立的推奨 |
| **HOLD** | スコープ固定。アーキテクチャ/セキュリティ/エッジケースに集中 |
| **REDUCTION** | 最小版。MVPのみ出荷、残りは次回 |

デフォルト判定:
- 新規機能 → EXPANSION
- 既存機能の改善 → SELECTIVE
- バグ修正/リファクタ → HOLD
- 15ファイル超 → REDUCTION提案

### 2E. Temporal Interrogation（EXPANSION/SELECTIVE/HOLDで実行）

```yaml
  HOUR 1 (基礎):     実装者が知るべきことは？
  HOUR 2-3 (核心):   どの曖昧さにぶつかるか？
  HOUR 4-5 (統合):   何に驚くか？
  HOUR 6+ (仕上げ):  何を事前に計画しておきたかったか？
```

## Step 3: Engレビュー（認知パターン15原則適用）

### 3-0. スコープチャレンジ

1. 既存コードが解決済みの部分はどこか？
2. 目標達成の最小変更セットは？
3. **8ファイル超え or 2新クラス超え → スメル警告**（※ フレームワーク特性やコンポーネント粒度により基準を調整可。例: マイクロサービス構成では1サービスあたりの許容ファイル数を緩和）
4. **Search Before Building**（Context7活用）:
   - `mcp_context7_resolve-library-id` → 使用ライブラリの最新ドキュメント確認
   - `mcp_context7_query-docs` → ベストプラクティス・定番パターン検索
   - フレームワーク組み込み機能の検索
5. **完全性チェック**: AI支援で限界コスト低→完全版を推奨

### 3-1. アーキテクチャレビュー（認知パターン適用）

評価項目:
- コンポーネント境界と依存グラフ
- データフロー（Happy/Nil/Empty/Error の4パス）
- 状態マシン
- スケーリング特性と単一障害点
- セキュリティアーキテクチャ
- 本番障害シナリオ（各統合点に1つ）
- ロールバック態勢

**Draw.io MCP でアーキテクチャ図を自動生成:**

```mermaid
mcp_drawio_open_drawio_mermaid:
  content: |
    graph TD
      subgraph [コンポーネント名]
        [ノード定義]
      end
```

**Context7 で技術選定を検証:**

```yaml
mcp_context7_resolve-library-id: [採用予定ライブラリ]
mcp_context7_query-docs: ["ベストプラクティス" or "既知の問題"]
```

**Boring by Default**: 実績のある技術か？イノベーショントークンを賢く使っているか？
**Reversibility**: Feature flag? canary? ロールバック手順は？

#### 主要原則 適用チェックリスト

| 原則 | 確認項目 | 判定 |
|------|----------|------|
| **Blast Radius** | 変更が影響する範囲は最小か？障害時の被害範囲は限定的か？ | ✓/✗ |
| **Boring by Default** | 実績のある技術選定か？新技術に正当な理由があるか？ | ✓/✗ |
| **Reversibility** | ロールバック手順は明確か？Feature flagで段階的に展開可能か？ | ✓/✗ |

### 3-2. エラー＆レスキューマップ

```python
METHOD/CODEPATH      | WHAT CAN GO WRONG    | EXCEPTION CLASS
---------------------|----------------------|----------------
                     |                      |

EXCEPTION CLASS      | RESCUED? | RESCUE ACTION    | USER SEES
---------------------|----------|------------------|----------
                     |          |                  |
```

GAP = RESCUED=N → **CRITICAL GAP**

### 3-3. セキュリティ＆脅威モデル

- 攻撃面拡大の評価
- 入力バリデーション（nil/empty/injection）
- 認可（IDOR脆弱性）
- シークレット管理
- 依存パッケージのセキュリティ

### 3-4. コード品質レビュー

- DRY違反（積極的にフラグ）
- 命名品質
- 循環的複雑度（5分岐超え → リファクタ提案）
- Over/Under-engineering

### 3-5. テストレビュー

```yaml
NEW UX FLOWS:    [各新インタラクション]
NEW DATA FLOWS:  [各新データパス]
NEW CODEPATHS:   [各新分岐/条件]
NEW ERROR PATHS: [各新エラーパス]
```

各項目:
- 何型テスト? (Unit/Integration/E2E)
- Happy path テスト ✓/✗
- Failure path テスト ✓/✗
- Edge case テスト ✓/✗

**金曜深夜テスト**: 金曜2AMに自信を持ってデプロイできるテストか？

### 3-6. パフォーマンスレビュー

- N+1クエリ
- メモリ使用量
- インデックス
- キャッシュ機会
- 遅いコードパスのp99レイテンシ推定

### 3-7. 可観測性＆デバッガビリティ

- ロギング（入口/出口/分岐で構造化ログ）
- メトリクス/アラート
- 3週間後のバグ→ログだけで再構築可能か？

### 3-8. デプロイ＆ロールアウト

- マイグレーション安全性（ゼロダウンタイム）
- Feature flag 推奨
- ロールバック手順（明示的ステップバイステップ）

## Step 4: Design UXレビュー（7パス評価）

UI/UXスコープがある場合のみ実行。

### Pass 1: 情報設計 (0-10)
ユーザーが最初・2番目・3番目に見るものは？Constraint worship — 3つだけ見せるなら？

### Pass 2: インタラクション状態カバレッジ (0-10)
```text
FEATURE          | LOADING | EMPTY | ERROR | SUCCESS | PARTIAL
-----------------|---------|-------|-------|---------|--------
```
空状態は機能 — 温かさ、主要アクション、コンテキスト。

### Pass 3: ユーザージャーニー＆感情弧 (0-10)
```text
STEP | USER DOES        | USER FEELS      | PLAN SPECIFIES?
-----|------------------|-----------------|----------------
```
時間軸設計: 5秒（直感）、5分（行動）、5年（関係）

### Pass 4: AIスロップリスク (0-10)
- 「カード + アイコン」→ 何が差別化か？
- 「ヒーローセクション」→ この製品らしさは？
- 「クリーンでモダン」→ 意味なし。具体的設計決定に置換

### Pass 5: デザインシステム整合性 (0-10)
DESIGN.mdがあれば照合。なければギャップフラグ。

### Pass 6: レスポンシブ＆アクセシビリティ (0-10)
- 各ビューポートで意図的なレイアウト
- キーボードナビ、スクリーンリーダー、タッチターゲット44px以上
- カラーコントラスト

### Pass 7: 未解決の設計決定
```text
DECISION NEEDED              | IF DEFERRED, WHAT HAPPENS
-----------------------------|---------------------------
空状態のデザインは？          | エンジニアが「アイテムなし」を出荷
モバイルナビパターンは？      | デスクトップナビがハンバーガーの裏に
```

## Step 5: 計画承認

全レビュー通過後、計画を `docs/plans/` に保存。

## AskUserQuestion ルール（4段構造）

```yaml
1. 再接地: 「現在の理解では [X] ですが、合っていますか？」
2. 平易化: 技術用語を避け、要点を1文で
3. 推奨:   「[Y] を推奨します。理由: [Z]」
4. 選択肢: 2-3の具体的選択肢（番号+文字でラベル、例: 3A, 3B）
```

ルール:
- **1問題 = 1質問**。バッチ禁止
- 「何もしない」を選択肢に含む
- **明白な修正は質問せず実行し報告**
